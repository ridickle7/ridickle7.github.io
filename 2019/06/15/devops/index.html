Part1 에자일 방법론과 DevOps

시간 : 14:10 ~ 14:50  |  발표자 : 배준현

#cloudInfra #BlueGreenDeploy #DevOps #AzureDevops

DevOps = Develop(개발) + Operation (운영)

Cloud Infra

  • AWS
  • GCP
  • AZure
  • KT Cloud…. (넘나 많은 것)

Cloud를 어디서 절실히 느끼게 될까?

노벨 평화상 사이트

  • 단 하루를 위해 존재하는 사이트
  • 평소 : 컴터 1대도 괜춘 / 투표일 : 전세계 수요가 한 방에 몰림
  • 이런 사이트를 구현할 때 클라우드는 개이득

무중단 배포

  • 블루그린 배포

    개발자들은 ‘Continuous (연속적) 하고 Automatic (자동화)’ 한 API 배포를 원함
    but 소프트웨어의 최종 테스트 단계에서 실제 운영 단계로 넘어갈 때의 ‘중단 시기‘ 존재
    동일하게 구성된 환경을 하나 더 추가함으로써 문제를 해결
    새로운 걸 하나 더 만들고 새버전 소스를 만듬

DevOps = Develop(개발) + Operation (운영)

  • 개발이랑 운영이랑 다 잘하자
  • Dev : Plan / Create / Verify / Package
  • Ops : Release / Configure / Monitor
  • github MD 파일만 수정을 했는데 알아서 컴파일 등등 다하고 홈페이지 바뀐 내용이 반영됨
  • 처음에 빡세게 구성하고 만들어놓으면, 그 이후로는 편~~~~하다.

DevOps가 없다면

  • 코드 만들고 -> 서버로 옮겨주세요 하고 -> 서버 올리고 -> 그 와중에 서버 오류 나고 #@$%^&%^$%@#$%$&^@$#

devops tools

  • chef, travis, jenkins, buddy/build 등등

devops 로직 example

  • github 의 pull request 된 거를 체크해서 코드를 다 가져옴
  • 실행 및 컴파일 되는지 확인
  • 테스트 되는지 확인
  • 도커로 박스를 만들어 AWS 서버로 보냄

    (아직까진 무슨 말인지 감이 덜 온다 / devops 구성하는데 8시간 걸렸다 카더라)

devops 사용 example

  • 사용자가 무엇을 클릭하면, 메일을 보낸다 (QA팀에서 요긴하게 사용)

에자일

  • 모듈을 만들어야하는데 어떻하지 -> 정말 필요한거에요? -> ㅇㅇ 살려줘 -> (한숨) 해야죠… -> 언제까지 가능? -> … (답변을 못함)
  • 하나하나 쪼개서 물어보면 작업이 좀 더 빠르게 진행되고 Task에 대한 명확한 답변이 가능해진다

    ex> 언제까지 가능? -> 2시간이요 -> (힘들 것 같은데…) 6시간 줄게

  • 후배 개발자들이 지정된 시간에 일을 완수하지 못했을 경우

    갈군다 (X)
    코드 리뷰를 해주고 레퍼런스를 제공해준다 (O)

연사님 story

  1. 프로그래밍 숙지사항

    • PreProc >> Read >> Compute >> Write 로 구성 (>> 에 TDD가 있다)

  2. 요구사항

    • 바꿀 수 없는 내용들을 미리 적어놓음 (DB 다 뒤집어 엎는 과정이 필요하다거나 등등)
    • AGile Splint

  3. todo / doing / done

    • 카드 형태로 올리고 개발자들이 알아서 가져감 (카드 내용 : 3시간 / ‘업무내용’)

  4. MySQLB

    • DESC 기본 지원 (indexing이 빨라짐)
    • json Type Column 제공

Part2 SW개발자를 위한 시스템 명령어 사용법

시간 : 15:00 ~ 15:40  |  발표자 : 김종일

#리눅스명령어

man만 알면 다 할 수 있어

  • 다 나와있긴 하지만 조합하는 게 이슈

면접관 입장에서 면접을 보고 느낀 점

  • 10년차 이상을 한 개발자인데 기본을 모른다 (ex> 객체지향 질문에 답을 못함)
  • 로컬에서는 되는 데, 서버에 올리면 안된다.

    Application programs 부분만 보고 있다.
    아랫단은(하드웨어나 네트웤 구성 등) 모든 것이 다르다.

1. Process

  • 실행 상태중인 프로그램
  • CPU Usage (%의 의미 : CPUTime(시간)에 대한 점유율의 percent)

    막연하게 알지 않고 용어 자체의 의미부터 파악해야한다.

  • load

프로그램 실행 순서

  • 평상시에는 디스크에 있음
  • 읽어서 메모리에 올림
  • 실행 포인트가 다가오면 startPoint 에 올려 레지스터와 캐시에 올려 실행시킴

ps -ef

  • 프로세스가 execution 상태인지 확인하는 명령어
  • 프로그램이 안 될 때 제일 먼저 이걸 확인하는 걸 추천

단어 의미 확인
Process Control Block(PCB) : 특정한 프로세스를 관리할 필요가 있는 정보를 포함하는, 운영체제 커널의 자료구조

ps -aux

  • ps -ef보다 많은 정보를 보여줌
  • 사용하는 스택의 크기, 사용된 메모리 등의 정보를 볼 수 있음

pstree

  • 내가 만든 프로세스가 또다른 프로세스를 실행시킬 수 있음

    프로그램을 잘못 짰을 때

  • process 간의 계층을 확인할 수 있음

2. Network

nmap -PN daum.net -p80 | grep^80

  • daum.net에서 80포트에 대해서 서비스를 운영하고 있는거야?
  • db 프로세스가 살아있는지 확인 -> port

nmap localhost

  • port scanner
  • 이 ip가 어떤 서비스를 하고 있는지 다 scan 가능

netstat -tnlp

  • listener with PID
  • 어떤 포트가 어떤 걸 물고 있는지 확인

nc -zv localhost 22

  • Simple connection check

iptables -nL –line-numbers

  • 혹시 방화벽 정책으로 막혀있지 않은가?

3. Routing

route

  • 혹시 라우팅이 문제인가?
  • default라 써있을 경우 다 저기로 가는 것

netstat -nr

4. RESTAPI TEST

curl

httpie

  • curl에 비해 쉬움

5. DISK

df -h

  • disk check
  • 리눅스는 100%차도 접근은 할 수 있음
  • 로그를 따로 관리함 (로그가 용량을 많이 먹음)

watch

  • 좀 있어보임
  • watch –interval=2 ‘ls -alh syslog’

6. LOG CHECK

  • /var/log 에 존재
  • 로그만 잘 보면 모든 문제를 확인할 수 있다.

tail -10 syslog | grsp

7. LOG ROTATE

  • 로그로 서버를 다 채울 수 있음
  • 로그 설정 꼭 하세요!
  • 로그를 하나 찍는데 하나의 쓰레드를 씀

8. Sync/Async/Blocking/Non-Blocking

  • blocking 로직
    i/o 때문에 멈춰있음
  • non-blocking 로직
    i/o 때문에 멈춰있진 않음

프로그램을 짤 때 불필요하게 i/o를 많이 일으키지 말자
아무리 사양 좋은 서버로 프로그래밍을 하더라도 i/o를 무개념하게 쓰면 느려진다

9. CPU Performance를 보기 위한 좋은 툴 : Top

  • 1.0이면 딱 고만큼
  • io wait : io가 빈번하게 많으면 application은 아무일도 안함 -> 이 데이터 값이 클 수록 CPU는 놀고 있는 거

vmstat
iostat

#데브시스터즈 #Kubernetes

데브시스터즈에서는 게임 서버 개발과 테스트 자동화를 위해 Kubernetes를 도입하였습니다.
발표자료

Kubernetes의 윈도우 OS 지원 : 2017년 말에 시작 (기본 default 리눅스 기반에서 동작)
데브시스터즈의 시행착오 github issue

쿠버네티스에서 왜 container를 사용하는가?

  • 기존 : 라이브러리 세팅, 방화벽 세팅 이런 걸 그대로 진행 (하루 그냥 날라감)
  • 쿠버네티스 : 위의 모든 과정을 container로 포장해놓고 그대로 사용함으로써 시간을 절약함

introduce 쿠버네티스

  • 2014년에 공개된 오픈 소스 프로젝트
  • 컨테이너 플랫폼
  • 마이크로서비스 플랫폼
  • 이식성 높은 클라우드 플랫폼

쿠버네티스 identitity

  • Pod를 통한 컨테이너 그룹 형성
  • 자가 복구
  • 오토 스케일링
  • DNS 관리
  • 부하 분산
  • 롤링 업데이트 / 롤백
  • 리소스 모니터링 / 로깅

Window에서는 라이브러리를 커스텀해서 처리하는 것이 불가능
